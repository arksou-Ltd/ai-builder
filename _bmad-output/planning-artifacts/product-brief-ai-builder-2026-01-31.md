# Product Brief: ai-builder

<!-- Content will be appended sequentially through collaborative workflow steps -->

## Executive Summary

ai-builder 是一个面向非技术人员的 AI Native 开发平台，致力于实现"人人都是 AI Builder"的愿景。该平台通过工作流引导和对话式交互，让产品经理能够直接将需求转化为符合项目规范的高质量代码（80分水平），研发人员只需进行审核和优化（提升至95分以上），从而彻底改变传统软件开发中PM与研发之间的协作模式，将需求实现周期从一个月缩短到极致。

---

## Core Vision

### Problem Statement

在传统软件开发流程中，产品经理编写PRD后需要交给研发团队实现，整个周期通常需要一个月。即使PM尝试使用AI工具创建demo，由于缺乏技术背景，产出的代码无法遵循项目规范和架构要求，导致研发人员无法复用，不得不重新实现——甚至有时基于现有架构根本无法实现PM的设想。

### Problem Impact

- **时间成本高**：需求从提出到上线需要约一个月
- **资源浪费**：PM的demo无法复用，研发需要从零开始
- **沟通鸿沟**：PM不懂技术规范，研发难以理解PM意图
- **创新受阻**：漫长的开发周期抑制了快速试错和迭代

### Why Existing Solutions Fall Short

| 现有方案 | 局限性 |
|---------|--------|
| Devin.ai | 面向技术人员设计，非技术人员难以使用 |
| BMAD-METHOD | 流程完善但仍需专业技术人员操作 |
| 低代码平台 | 灵活性有限，难以适配复杂项目规范 |

**核心问题：现有AI开发工具都是为技术人员设计的，没有真正赋能非技术人员。**

### Proposed Solution

ai-builder 采用三层架构解决这一问题：

1. **工作流引导层**：将软件开发流程结构化，在每个节点提供对话式AI交互，引导非技术人员完成需求表达和实现

2. **项目规范层**：研发人员预先在GitHub建立符合规范的项目模板和文档，AI自动学习并遵循这些规范进行开发，PM无需关心技术细节

3. **协作优化层**：PM通过平台产出80分质量的代码，研发人员进行审核和优化至95分以上，实现高效分工

**用户体验**：产品经理只需授权GitHub，将项目添加到工作空间，即可开始通过对话实现需求。

### Key Differentiators

| 差异化优势 | 说明 |
|-----------|------|
| **非技术人员友好** | 专为PM等非技术角色设计，无需编程知识 |
| **项目规范自适应** | 自动遵循项目已有的架构和规范，代码可直接复用 |
| **80/95分工模式** | 重新定义PM与研发的协作边界，最大化双方价值 |
| **AI Native时代定位** | 顺应"人人都是Builder"的趋势，填补市场空白 |

**创始人优势**：专业技术背景 + 深厚项目管理经验，深刻理解两端痛点。

## Target Users

### Primary Users

#### 产品经理 Persona: 小李

**背景画像：**
- 在创业公司工作，3年以上产品经验
- 日常工作：撰写产品需求文档，交付给研发团队实现
- 有创新想法，渴望快速验证和落地

**当前痛点体验：**
- 尝试用 AI 工具（如 ChatGPT、Cursor）写 demo，成功实现了功能
- 兴奋地交给研发，却被告知"这个实现不了"
- 困惑：明明 demo 已经跑通了，为什么不能用？
- 真相：demo 代码不符合项目的技术架构和规范，无法复用
- 结果：研发从零开始重新实现，周期拉长到一个月

**成功愿景：**
- "哇塞，我可以直接实现80分的需求！"
- "研发不需要从头开始了，只需要优化升级！"
- 从"需求提出者"升级为"需求实现者"

**决策模式：**
- PM 自己决定是否采用工具
- 不需要复杂的采购流程

### Secondary Users

#### 研发人员 Persona: 小王

**背景画像：**

**价值获得：**
- 团队流程更通畅高效
- 聚焦于高价值的架构和优化工作
- 减少重复性的基础开发工作

## MVP Scope

### Core Features

#### 1. GitHub 授权与项目导入
- 通过 GitHub OAuth 登录（无需独立账号系统）
- 授权后可将 GitHub 仓库添加到工作空间
- 支持的项目类型：Flutter、Java (Spring)、Next.js

#### 2. AI 渠道配置
- 支持配置 AI 服务渠道：
  - **Claude Code**：URL + API Key
  - **Codex**：URL + API Key
- 用户可自行配置和切换 AI 渠道

#### 3. 工作流引导
PM 按以下步骤完成需求开发：

| 步骤 | 说明 |
|------|------|
| 1. 拉取代码 | 自动拉取最新代码 |
| 2. 创建分支 | 自动创建 feature 分支 |
| 3. 创建史诗 | 引导 PM 描述用户需求（史诗级别） |
| 4. 生成故事 | AI 根据史诗自动拆解为用户故事 |
| 5. 开发与校验 | 完成每个故事的代码实现和校验 |
| 6. 手工验证 | 运行验证确保功能正常 |
| 7. Push & PR | 自动提交代码并创建 PR |
| 8. 通过审核 | 研发 Review 并合并 |

*工作流设计可参考 BMAD-METHOD 的最佳实践*

#### 4. 对话式需求实现
- 在每个工作流节点中，PM 可通过对话与 AI 沟通
- AI 理解需求后生成符合项目规范的代码
- 支持多轮对话澄清和调整

#### 5. 代码生成与 PR 提交
- AI 生成符合项目架构和规范的代码
- 自动处理 Git 操作（commit、push）
- 自动创建 Pull Request

### Out of Scope for MVP

| 功能 | 状态 | 说明 |
|------|------|------|
| 独立账号系统 | ❌ 不做 | 仅使用 GitHub 登录 |
| 付费功能 | ❌ 不做 | MVP 阶段免费使用 |
| 多人协作/团队功能 | ❌ 不做 | 后续根据反馈考虑 |
| 后台管理系统 | ❌ 不做 | MVP 无需管理后台 |
| 多项目管理 | ⏸️ 延后 | 根据用户反馈优化 |
| 高级报表/数据分析 | ⏸️ 延后 | 根据用户反馈优化 |
| 其他项目类型支持 | ⏸️ 延后 | 先聚焦 Flutter/Spring/Next.js |
| 更多 AI 渠道 | ⏸️ 延后 | 先支持 Claude Code 和 Codex |

---

### MVP Success Criteria

**核心验证标准：**
> **有一个需求成功通过 PR 审核**

这意味着：
- PM 完整走通了工作流的所有步骤
- AI 生成的代码符合项目规范
- 研发 Review 后确认代码可用
- 验证了"PM产出80分代码"的核心假设

**进入下一阶段的信号：**
- 首个需求成功通过 → 继续验证更多需求
- 累计20个需求完成 + 20%一次性通过率 → 开始扩展用户
